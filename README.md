# ETL-процессы и хранилище данных (DWH)

Учебный мини-проект: разбор того, как данные из операционной системы попадают
в хранилище данных и превращаются в аналитику. На одном сквозном сценарии —
**продажи интернет-магазина** — показан полный конвейер **ETL** (Extract →
Transform → Load) и устройство витрины по схеме **«звезда»**. Сервисы можно
поднять локально и руками увидеть, как сырые операционные данные превращаются
в отчёты.

## OLTP vs DWH: зачем вообще два хранилища

Бизнес-приложение (интернет-магазин, банк, склад) работает с **операционной
БД (OLTP** — Online Transaction Processing). Она заточена под много мелких
быстрых транзакций: оформить заказ, списать товар, обновить статус. Данные
нормализованы (разложены по множеству таблиц без дублирования), чтобы запись
была быстрой и согласованной.

Для аналитики такая база неудобна: «выручка по категориям за квартал» — это
тяжёлый запрос с кучей JOIN-ов, который вдобавок мешает рабочей нагрузке. Поэтому
заводят отдельное **хранилище данных (DWH** — Data Warehouse): сюда регулярно
переливают данные из источников, перекладывают в удобную для чтения форму и
уже по нему строят отчёты и дашборды.

|                     | OLTP (источник)                | DWH (хранилище)                  |
| ------------------- | ------------------------------ | -------------------------------- |
| Назначение          | транзакции приложения          | аналитика и отчётность           |
| Нагрузка            | много мелких чтений/записей    | редкие тяжёлые запросы на чтение |
| Модель данных       | нормализованная (3НФ)          | денормализованная («звезда»)     |
| Свежесть данных     | здесь и сейчас                 | с задержкой (после загрузки ETL) |
| Пример              | таблицы `orders`, `order_items`| факт `fact_sales` + измерения    |

## Что такое ETL

**ETL** — это процесс переноса данных из источников в хранилище в три шага:

```
   ИСТОЧНИК(и)            ETL-конвейер                       DWH
 ┌─────────────┐   ┌───────────────────────────┐    ┌──────────────────┐
 │  OLTP-база  │   │ Extract → Transform → Load │    │  схема «звезда»  │
 │  магазина   │──▶│  тянем    чистим/   пишем   │───▶│  факты+измерения │──▶ отчёты
 └─────────────┘   │           считаем  в витрину│    └──────────────────┘
                   └───────────────────────────┘
```

- **Extract (извлечение).** Забираем сырые данные из источников. Источником может
  быть БД, API, файлы, очередь. Здесь данные читаются как есть.
- **Transform (преобразование).** Самый содержательный шаг: чистка, приведение
  типов и форматов, объединение из разных источников, расчёт показателей
  (мер), построение измерений и проставление суррогатных ключей.
- **Load (загрузка).** Складываем готовые таблицы в DWH — как правило, измерения,
  затем факты.

> **ETL vs ELT.** В классическом ETL преобразование происходит *до* загрузки
> (в отдельном движке конвейера). В современном ELT сырьё сначала грузят в
> мощное хранилище, а преобразуют уже *внутри* него (SQL/dbt). В этом проекте
> реализован именно **ETL**: преобразование живёт в `etl-service` до записи в DWH.

## Схема «звезда» (star schema)

В DWH данные раскладывают в **звезду**: в центре — таблица-**факт** с числовыми
показателями события, вокруг — таблицы-**измерения** (справочники), по которым
эти показатели «режут».

```
        dim_customer                 dim_date
        (кто купил)                 (когда купили)
              \                        /
               \                      /
                \                    /
                 ┌──────────────────┐
                 │    fact_sales     │   ← числа: quantity, amount
                 │  (факт продажи)   │
                 └──────────────────┘
                          |
                          |
                     dim_product
                     (что купили)
```

- **Факт** (`fact_sales`) — одна строка на проданную позицию: меры `quantity`,
  `unit_price`, `amount` и внешние ключи на измерения.
- **Измерения** (`dim_customer`, `dim_product`, `dim_date`) — контекст: кто, что,
  когда. По ним группируют и фильтруют.
- **Меры (measures)** — то, что суммируют (выручка, количество).

### Суррогатные ключи

В измерениях два ключа: **natural key** (`customer_id` — пришёл из источника) и
**surrogate key** (`customer_key` — собственный автоинкрементный ключ DWH).
Факт ссылается на *суррогатный* ключ. Это развязывает хранилище от источника:
можно менять источник, переиспользовать измерения, версионировать историю
(slowly changing dimensions) — не ломая факты.

## Принцип «database per service»

У каждого сервиса — **своя БД**, и чужие таблицы напрямую никто не читает,
только через API. Источник владеет своей операционной БД, DWH — своей витриной.
Шаг Extract обращается к источнику **по HTTP**, а не лезет в его таблицы — ровно
как в реальной интеграции.

В этом проекте обе БД — **локальные SQLite-файлы** (через SQLAlchemy):
`source.db` у источника, `dwh.db` у витрины. SQLite выбран ради нулевой
инфраструктуры — показать архитектуру, а не возиться с настройкой. В проде это
PostgreSQL/ClickHouse/Snowflake/BigQuery: меняется одной строкой `DATABASE_URL`,
код не трогаем.

## Что делает каждый сервис

### `source-service` — операционная система (OLTP)

Имитирует бэкенд интернет-магазина. Владеет своей БД `source.db` с
нормализованными таблицами `customers`, `products`, `orders`, `order_items`,
`returns` и при старте наполняет её детерминированными сид-данными. Наружу
отдаёт данные только через API:

| Эндпоинт      | Что отдаёт                                          |
| ------------- | --------------------------------------------------- |
| `GET /customers` | справочник клиентов                              |
| `GET /products`  | справочник товаров                               |
| `GET /sales`     | плоский поток позиций заказов (сырьё для продаж) |
| `GET /returns`   | возвраты (используются в лабораторной работе)    |

### `etl-service` — конвейер + витрина DWH

Владеет своей БД `dwh.db` (схема «звезда»). Запускает конвейер и поверх витрины
отдаёт аналитику:

| Эндпоинт                         | Что делает                                       |
| -------------------------------- | ------------------------------------------------ |
| `POST /etl/run`                  | полный прогон Extract→Transform→Load             |
| `GET /dwh/stats`                 | число строк в каждой таблице витрины             |
| `GET /reports/sales-by-category` | выручка по категориям товаров                    |
| `GET /reports/sales-by-month`    | выручка по месяцам                               |
| `GET /reports/top-customers`     | клиенты по сумме покупок                         |

Логика конвейера разнесена по модулям: `etl.py` — Extract/Transform/Load,
`db.py` — модели звезды, `main.py` — API и отчёты.

## Настройки

Никаких «магических чисел» в коде — всё через переменные окружения (в коде
только дефолты), задаются в `docker-compose.yml`.

| Переменная     | Сервис | По умолчанию              | Что задаёт                       |
| -------------- | ------ | ------------------------- | -------------------------------- |
| `DATABASE_URL` | оба    | `sqlite:///./<сервис>.db` | строка подключения к своей БД     |
| `SOURCE_URL`   | etl    | `http://source:8000`      | адрес источника для шага Extract |
| `HTTP_TIMEOUT` | etl    | `10.0`                    | таймаут HTTP-вызовов, сек        |

Захочешь перевести сервис на свой PostgreSQL — пропиши
`DATABASE_URL=postgresql+psycopg2://...` (нужен драйвер `psycopg2`), код менять
не нужно.

## Как запустить

Нужен **Docker** и **Docker Compose**. Сервисы на Python (FastAPI), у каждого —
своя локальная БД (SQLite). `docker compose up` поднимет всё сам.

```bash
docker compose up --build
```

Запускаем конвейер (в другом терминале):

```bash
curl -X POST localhost:8000/etl/run
# -> {"status":"OK","loaded":{"dim_customer":4,"dim_product":4,"dim_date":5,"fact_sales":8}}
```

Смотрим, что получилось в витрине:

```bash
curl localhost:8000/dwh/stats
# -> {"dim_customer":4,"dim_product":4,"dim_date":5,"fact_sales":8}

curl localhost:8000/reports/sales-by-category
# -> {"Электроника":124500.0,"Книги":3200.0,"Продукты":7000.0}

curl localhost:8000/reports/top-customers
# -> {"Иван":123000.0,"Anna":6500.0,"John":2800.0,"Олег":2400.0}
```

Источник тоже доступен снаружи (порт `8001`) — можно посмотреть сырьё:

```bash
curl localhost:8001/sales
```

Остановить: `Ctrl+C`, затем `docker compose down`.

## Что важно заметить

1. **Два контура данных.** Источник нормализован под транзакции; витрина —
   денормализована под чтение. Это разные модели для разных задач.
2. **Конвейер идемпотентен.** `POST /etl/run` каждый раз пересобирает витрину
   с нуля (full reload) — повторный запуск даёт тот же результат.
3. **Extract идёт через API источника**, а не напрямую в его БД — это и есть
   «database per service» на практике.
4. **Суррогатные ключи** в измерениях независимы от ключей источника — факт
   ссылается именно на них.

## Структура репозитория

Каждый сервис — самостоятельный проект со своим `Dockerfile` и
`requirements.txt`; `docker compose` собирает каждый из своей папки.

```
etl_dwh/
├── README.md
├── docker-compose.yml
├── source-service/        # OLTP-источник (своя БД source.db)
│   ├── db.py              #   модели + сид операционных данных
│   ├── main.py            #   API: /customers /products /sales /returns
│   ├── Dockerfile
│   └── requirements.txt
├── etl-service/           # конвейер + витрина DWH (своя БД dwh.db)
│   ├── db.py              #   схема «звезда» (измерения + факт)
│   ├── etl.py             #   Extract / Transform / Load
│   ├── main.py            #   API: /etl/run /dwh/stats /reports/*
│   ├── Dockerfile
│   └── requirements.txt
└── lab/                   # лабораторная работа
    ├── LAB.md             #   задание, контракт, критерии, рубрика
    └── autotest/          #   самопроверяемый автотест по HTTP
        ├── autotest.py
        ├── requirements.txt
        └── README.md
```

> Бизнес-логика упрощена (full reload вместо инкрементальной загрузки, без
> ретраев и идемпотентности на уровне строк, БД — локальный SQLite): цель —
> показать устройство ETL и схему «звезда», а не продакшен.

## Лабораторная работа

В папке `lab/` — задание: **провести через конвейер новое бизнес-событие
«возврат товара»**. Источник уже отдаёт возвраты (`GET /returns`), но витрина их
не учитывает. Нужно добавить факт `fact_returns` в схему, обработать возвраты на
всех трёх шагах ETL и сделать отчёт по возвратам. Так на практике виден полный
путь данных: **источник → факт в звезде → отчёт**, включая работу с измерениями
и суррогатными ключами.

- Задание целиком — в `lab/LAB.md` (цель, контракт таблицы-факта, шаги,
  критерии приёмки, рубрика).
- Лаба **самопроверяемая**: автотест `lab/autotest/` сам гоняет конвейер по HTTP
  и проверяет витрину и отчёты. Базовые проверки (продажи) проходят сразу,
  проверки по возвратам — после выполнения задания. Как запускать — в
  `lab/autotest/README.md`.
