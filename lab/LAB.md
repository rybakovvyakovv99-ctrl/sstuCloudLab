# Лабораторная работа: добавить факт «Возвраты» в DWH

## Цель

Научиться расширять ETL-конвейер и хранилище данных новым бизнес-событием.
Источник (`source-service`) уже отдаёт данные о возвратах товаров
(`GET /returns`), но конвейер их не обрабатывает, и в витрине их нет. Нужно
провести возвраты через все три шага ETL и сделать аналитический отчёт.

Так на практике видно полный путь данных: **источник → факт в звезде → отчёт**,
включая работу с измерениями и суррогатными ключами.

## Что уже есть (трогать не нужно)

- `source-service` отдаёт возвраты: `GET /returns` →
  `[{return_id, order_item_id, quantity, reason, created_at}, ...]`.
- Базовый конвейер по продажам полностью работает: измерения
  `dim_customer`, `dim_product`, `dim_date` и факт `fact_sales`.
- Отчёты по продажам: `/reports/sales-by-category`, `/sales-by-month`,
  `/top-customers`.

## Что нужно сделать

Возврат привязан к позиции заказа (`order_item_id`). Товар, клиента и цену по
этому `order_item_id` можно найти в потоке продаж (`GET /sales`, поле
`order_item_id`). Дата факта возврата — это `created_at` самого возврата
(а не дата заказа!).

### 1. Модель факта в DWH — `etl-service/db.py`

Объявить таблицу-факт `fact_returns` (класс `FactReturns`) со ссылками на
существующие измерения. Минимальный контракт:

| Поле           | Тип     | Смысл                                  |
| -------------- | ------- | -------------------------------------- |
| `return_id`    | PK      | = `return_id` источника                |
| `date_key`     | FK→dim_date     | дата возврата                  |
| `customer_key` | FK→dim_customer | клиент (из исходной позиции)   |
| `product_key`  | FK→dim_product  | товар (из исходной позиции)    |
| `quantity`     | int     | сколько единиц вернули                 |
| `amount`       | float   | сумма возврата = `quantity * unit_price` |

Новая модель автоматически попадёт в `Base.metadata`, а значит и в `/dwh/stats`.

### 2. Обработка в конвейере — `etl-service/etl.py`

- **Extract:** дотянуть `GET /returns` из источника.
- **Transform:** по `order_item_id` найти товар, клиента и `unit_price` из
  потока продаж (удобно построить индекс `order_item_id → строка продажи`);
  посчитать `amount`. **Важно:** для даты возврата вызвать `ensure_date(...)` —
  даты возвратов тоже должны попасть в `dim_date`.
- **Load:** записать строки в `fact_returns`.
- Добавить количество загруженных возвратов в возвращаемые счётчики под ключом
  `fact_returns`.

### 3. Отчёт — `etl-service/main.py`

Добавить эндпоинт `GET /reports/returns-by-category`, возвращающий сумму
возвратов по категориям товаров: `{"Электроника": ..., ...}` (по аналогии с
`sales-by-category`, но join к `fact_returns`).

## Критерии приёмки

После `POST /etl/run` должно выполняться:

- `GET /dwh/stats` содержит `fact_returns` = **3**, а `dim_date` = **8**
  (5 дат заказов + 3 даты возвратов).
- `GET /reports/returns-by-category` возвращает:
  `Электроника = 1500`, `Продукты = 1000`, `Книги = 800`.

Всё это проверяет автотест в `autotest/` — см. `autotest/README.md`.
Лаба сдана, когда автотест проходит полностью (и базовые проверки, и проверки
по возвратам).

## Рубрика (10 баллов)

| Критерий                                                        | Баллы |
| --------------------------------------------------------------- | ----- |
| Модель `fact_returns` объявлена корректно (FK на измерения)     | 2     |
| Extract возвратов из источника                                  | 1     |
| Transform: верный lookup товара/клиента/цены, расчёт `amount`   | 2     |
| Даты возвратов попадают в `dim_date` (`dim_date` = 8)           | 1     |
| Load в `fact_returns` (stats = 3)                               | 1     |
| Отчёт `/reports/returns-by-category` с верными числами          | 2     |
| Автотест проходит полностью                                     | 1     |

## Подсказки

- Суррогатные ключи клиентов/товаров уже построены при загрузке измерений —
  переиспользуйте ту же логику (`customer_id → customer_key` и т.п.).
- `full reload` в начале `run()` дропает витрину целиком — ваш `fact_returns`
  пересоздастся автоматически, отдельной миграции не нужно.
- Если `dim_date` получился = 5, значит вы не прогнали даты возвратов через
  `ensure_date(...)`.
